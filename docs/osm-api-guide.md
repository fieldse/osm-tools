# OpenSourceMalware.com as a Threat-Intelligence Feed for Software Supply-Chain Security: An Evaluation

## TL;DR
- **OpenSourceMalware.com (OSM) is a credible, community-verified, malicious-package-focused feed worth adding as an early-warning layer** — it covers 11+ package ecosystems plus repos, containers, domains, IPs and crypto wallets, exposes a clean REST/JSON API and a STIX 2.1 option, and is run by recognized supply-chain researcher Paul McCarty. Its free tier (60 req/min, GitHub sign-up) is enough to pilot a CI/CD `check-malicious` gate immediately.
- **The catch is that the most pipeline-useful endpoints are paywalled and behind an org-only, contact-us-priced "Researcher Pro/Enterprise" tier.** The bulk polling feed (`/threat-feed`), full IOCs (`/threat-data`), STIX bundles, and the Anomali feed all require Pro; the free tier only confirms maliciousness and returns light metadata. Data is delivered in a **custom JSON schema, not OSV** — so it will not drop natively into OSV-Scanner/Grype and needs a normalization shim.
- **Treat OSM as a complement, not a replacement, for OSV Scanner, Grype + OSSF Malicious Packages, and Socket.dev.** Its differentiated value is human-verified campaign/threat-actor tagging and IOCs (C2 domains, hashes, wallets) that OSV malicious-package records lack, and McCarty's track record of publishing on fast-moving incidents before major vendors. Its weaknesses are no published false-positive rate, a documented human-review latency of hours, and the fact that it itself aggregates OSV/GHSA/Socket, so it is not a strictly independent source.

## Key Findings

**What it is.** OSM is a community-driven threat-intelligence database for "malicious open source" founded by Paul McCarty (alias "6mile"), a supply-chain security researcher who also serves as Head of Research at Safety CLI and teaches supply-chain red-teaming at BSides. The site bills itself as a free feed and API providing "a threat description, payload, and version details, and evidence" for malicious resources across the open-source supply chain. McCarty has described it as "the world's largest open database for software supply chain threat intel" (BankInfoSecurity).

**Coverage.** OSM tracks three asset categories in one unified DB:
- **Packages**: npm, PyPI, Maven, NuGet, RubyGems, Packagist, Crates.io, Go Modules, VS Code Marketplace, Open VSX, and AI "Skills" registries.
- **Attacker infrastructure**: domains/URLs (C2, phishing, exfil), IP addresses, and crypto wallets (including "ether hiding"/blockchain-steganography wallets).
- **Repositories & containers**: GitHub/GitLab/Bitbucket repos; Docker Hub, GHCR, and Quay images.

This is broader than OSV's malicious-package scope (which is package-only) — OSM deliberately captures the IOCs (IPs, domains, hashes, wallets, publisher emails, GitHub accounts) that CTI teams can action but that OSV/GHSA malicious-package records typically omit.

**Data sourcing & freshness.** Per a Feedly/Paul McCarty CTI session, OSM **aggregates data from GHSA, OSV, and security research sites like Socket.dev, then enriches it** with CTI-relevant metadata (campaign tags, threat-actor linkage, impact assessment, IOCs). This is important: OSM is *partly downstream* of OSV/GHSA, so it is not a fully independent corroborating source for those records. However, McCarty and a cross-company intel-sharing group also originate first-hand reports on live incidents — e.g., OSM published threat reports/advisories on the March 2026 Trivy/`trivy-action` compromise and the "ClawHavoc"/ClawHub malicious-skills campaign, in some cases ahead of major vendors. So freshness is mixed: some entries lag OSV (because they're ingested from it); others are published faster than OSV/GHSA because they come from McCarty's own research and an informal researcher network.

**Signal quality / verification.** Every report passes a 4-stage human review pipeline (Pending → In Review → Under Verification → Verified) with documented review criteria (evidence quality, technical accuracy, completeness, uniqueness) and a published timeline (assignment 1–6h; initial review 6–24h; final verification 24–48h). High-severity items get a second independent reviewer. Outcomes include "Rejected (false positive)," and the docs state false positives are tracked on contributor profiles. The docs concede "while false positives are still possible, this process helps us keep them to a minimum" — but **no quantitative false-positive rate is published.** A `last_osm_score` (0–100) and `confidence_level` per IOC provide some signal grading.

**Data format / normalization.** The default API is a **custom JSON schema** (fields like `malicious`, `report_type`, `severity_level`, `last_osm_score`, `details.threat_id`, `iocs[]`). It is **not** OSV format and **not** PURL-keyed, so it will not flow natively into OSV-Scanner, Grype, or the OSSF Malicious Packages store, all of which are OSV/PURL-native. OSM does offer **STIX 2.1** output on paid tiers — both a legacy indicator-only bundle and a `format=v2` bundle (identity + TLP:WHITE marking + an `x-malicious-package` custom object + indicator SDO + relationship SRO) purpose-built for Anomali ThreatStream; STIX patterns use CPE notation, not PURL. Interestingly, STIX records carry `external_references` back to OSV (e.g., an OSV URL), which helps de-dupe against an OSV-keyed pipeline.

**Pricing / access tiers.**
- **Free tier**: self-service via GitHub OAuth at opensourcemalware.com/auth (email accounts on request); generate an `osm_`-prefixed Bearer token under Settings → API Tokens. 60 requests/minute. Free endpoints: `GET /check-malicious`, `GET /query-latest` (100 most recent verified threats per ecosystem), `POST /submit-threat`, `PUT /modify-threat-report`.
- **Researcher Pro / Enterprise**: 180 req/min; unlocks `GET /threat-feed` (poll verified threats in a rolling 1–24h window), `GET /threat-data` (full record + all IOCs by `threat_id`), `GET /query-by-username` (all threats by a publisher), `GET /query-latest-stix`, and `GET /anomali-feed`. **Paid accounts are organization-only ("not available for individuals") and pricing is "contact us" — no published dollar figures anywhere.** STIX access is gated behind emailing info@osmsecurity.com.

**Licensing.** OSM's docs say data may be used "for non-commercial purposes," which initially reads as barring corporate use — but the docs explicitly resolve this: "non-commercial" means *internal* use, not resale. A for-profit company may use the free API/feed to protect its own org, including "Feed data into your internal tools, processes, or pipelines," automated lookups in security workflows (explicitly "encouraged"), and even bulk retroactive environment sweeps during an incident ("permitted for internal use"). The one explicit, all-tier prohibition: "no user may harvest or stockpile data beyond what the API is designed to provide. This includes the use of automated tools (bots, scrapers, crawlers)." Reselling/redistributing the data or building a competing commercial product on it is the implied "commercial" line (full Terms of Use are on a JavaScript-rendered page that could not be machine-fetched, so confirm the exact prohibited-use clauses in a browser before deployment).

**Integrations & tooling.** There is **no first-party SDK or client library** — integration is raw REST (curl/PowerShell/fetch examples in the docs). The docs provide a JavaScript "check-and-enrich" reference function and patterns for SIEM/SOAR/TIP ingestion (OpenCTI via STIX, Anomali ThreatStream via the custom feed). OSM also publishes an RSS feed (opensourcemalware.com/rss.xml, carried in Feedly's Cybersecurity bundle), a public GitHub org (incident-response repos, not a client SDK), a podcast, and McCarty's personal GitHub (github.com/6mile). I found no evidence of a widely adopted third-party OSM client or a Grype/OSV-Scanner plugin as of May 2026.

## Details

**The detection-gap thesis OSM is built on.** McCarty's framing (Feedly CTI session, Dec 2025) is that SCA/SAST tools look for known CVEs in legitimate libraries, while EDR is tuned for binary malware and often has developer-machine exclusions — leaving a gap that intentionally malicious packages drive straight through. OSM is purpose-built for that gap: interpreted-language malware has no stable binary hash, payloads iterate (sometimes multiple versions/day), and removal from registries is slow (he cites disclosing `Express.exp` to npm in Dec 2024, removed only in May 2025 — a five-and-a-half-month exposure, per his own account; I could not independently verify these specific dates). NPM dominates: per Sonatype's "2024 in Open Source Malware" report (Dec 10, 2024), "npm accounted for 98.5% of the malicious packages identified by Sonatype over the course of 2024," with PyPI at ~1% (over 540,000 malicious components found on npm vs. ~5,000 on PyPI). For context on scale, Sonatype's Q2 2025 Open Source Malware Index (July 8, 2025) states "the total number of open source malware packages Sonatype has discovered" reached 845,204 cumulatively — a 188% year-over-year surge, with 16,279 new packages found in Q2 2025 alone; Checkmarx separately claims 420,000+ in its database. Any single feed, OSM included, sees only a slice, which is why layering feeds matters.

**`check-malicious` (the workhorse, free).** A single GET with `report_type` (`package|container|repository|url|domain|ip|wallet`), `resource_identifier`, optional `ecosystem` and `version`. Returns `malicious: true/false` with `severity_level`, `description`, `tags`, `first_seen`/`last_seen`, `last_osm_score`, `last_scanned_at`, `scan_count`, and a `details.threat_id`. The `last_scanned_at` field lets you reason about staleness (docs suggest treating a scan within 7 days as fresh for most teams, 24h for high-security). This endpoint alone supports a pre-install/CI gate or an AI-coding-assistant guardrail.

**`threat-data` (full IOCs, paid).** Takes the `threat_id` from a check and returns the complete record with an `iocs[]` array — each IOC typed (`domain`, `c2_server`, `url`, `file_hash_sha256`, `email_address`, `wallet`, etc.), with `confidence_level` and a description. The worked example (`latinum-wallet-mcp`, a PyPI Solana wallet-stealer) returns the C2 exfil domain, payload SHA-256, and publisher email — exactly the actionable IOCs for firewall/EDR/SIEM rules. This is the two-step "check → enrich" workflow OSM recommends for SOAR.

**`threat-feed` (bulk polling, paid).** Poll per-ecosystem with `hours=1..24`; returns verified threats in the window, deduplicated by `id` (via `verified_at`). Designed for hourly cron polling to keep blocklists current and never miss a threat, with a catch-up window after poller outages. This is the endpoint you'd actually wire into a TIP/blocklist pipeline — and it's Pro-only.

**STIX details.** `query-latest-stix?ecosystem=npm&format=v2` returns a TLP:WHITE-marked bundle attributed to an "OpenSourceMalware" identity, with all indicators at confidence 100 and patterns in `[software:name = '…' AND software:cpe = '…']` form. Useful for OpenCTI/Anomali, but the CPE (not PURL) pattern means correlation against a PURL-keyed dependency graph requires mapping.

**Independence caveat for corroboration.** Because OSM ingests OSV/GHSA/Socket, using it to "corroborate" an OSV finding is partly circular. Its genuinely independent value is (1) net-new first-party reports from McCarty's research/intel-sharing group, (2) the enrichment layer (campaign tags, threat-actor linkage, IOCs) that OSV malicious-package records don't carry, and (3) asset types OSV doesn't track at all (C2 domains/IPs, wallets, containers, repos). OSM is also notably active on 2026 campaigns, often with same-day advisories. Worth calibrating the scale of these campaigns against multi-vendor reporting:
- **Shai-Hulud** (npm worm) — CISA's Sept 23, 2025 alert cited the original worm compromising "over 500 packages"; Datadog Security Labs reported Shai-Hulud 2.0 (Nov 24, 2025) "successfully taken over and backdoored 796 unique npm packages" across 1,092 versions with 20M+ combined weekly downloads. (OSM's own campaign tag surfaced ~200 related packages at the time of McCarty's demo.)
- **TeamPCP / "Mini Shai-Hulud"** — per Snyk's May 2026 analysis, the wave "published more than 300 malicious package versions across 323 packages in a 22-minute automated burst on May 19, 2026," from a compromised `atool` npm account maintaining 547 packages (~16M weekly downloads).
- **GlassWorm** — Socket's March 13, 2026 report documented "at least 72 additional malicious Open VSX extensions since January 31, 2026"; Aikido/Koi attribute the broader campaign to 151 infected GitHub repositories (Koi's initial Oct 2025 wave saw 7 extensions with ~36,000 downloads). OSM's follow-up analysis put affected projects/packages at 433+ across GitHub Python/JS repos, VS Code extensions, and npm.

## Recommendations

**Stage 1 — Pilot now (free, ~1–2 days of work).**
1. Create a free GitHub-auth account, mint an `osm_` token, store it as a CI secret.
2. Add a non-blocking `check-malicious` step to CI that queries each *direct* dependency (dedupe first; cache responses ~24h to respect the 60 req/min limit). Log hits; don't fail builds yet. Pair `report_type=package&ecosystem=…`.
3. Poll `query-latest` (free) per ecosystem you depend on (npm/PyPI first) into a dashboard or Slack alert to gauge signal volume and overlap with what OSV-Scanner/Grype already catch.
4. Subscribe to the RSS feed and podcast for campaign-level awareness.

**Stage 2 — Decide on paid (gate on pilot metrics).** Move to Researcher Pro/Enterprise only if the pilot shows OSM surfacing malicious packages **before** they appear in your OSV/GHSA-fed tooling, or surfacing IOCs you can't otherwise get. The Pro features you're buying are `/threat-feed` (real pipeline ingestion), `/threat-data` (IOCs for blocking), STIX/Anomali, and 180 req/min. Because pricing is contact-us and org-only, request a quote and negotiate against the concrete value of faster time-to-detection. Benchmark to watch: median lead time (hours/days) by which OSM-verified npm/PyPI threats precede your existing feeds, and number of actionable IOCs per incident.

**Stage 3 — Architect the integration.** Treat OSM as an enrichment/early-warning source layered on an OSV-native core:
- Keep OSV-Scanner + Grype (with the OSSF malicious-packages shell check) as the OSV/PURL-keyed backbone and CI gate.
- Write a thin normalizer mapping OSM JSON → your internal schema (or OSV-like records), keyed by ecosystem+package(+version); use the STIX `external_references` OSV URLs to de-dupe against OSV records.
- Route OSM IOCs (domains/IPs/hashes/wallets) to your SIEM/EDR/firewall via the STIX/Anomali path or the `threat-data` JSON.
- Use Socket.dev for behavioral/real-time install-time analysis (a different signal than OSM's verified-DB model) — the three are complementary, not redundant.

**What would change the recommendation:** If OSM published a quantified false-positive rate and the pilot showed low FPs, you could promote `check-malicious` from logging to a *blocking* CI gate. If OSV/GHSA latency for malicious packages shrinks to near-real-time, OSM's timeliness edge narrows and the free tier may suffice. If you need a strictly independent second source, weight OSM's *first-party* reports over its ingested-from-OSV entries.

## Caveats
- **No published false-positive rate.** Verification is human and documented, but FP quality is asserted, not quantified. Human review also introduces hours-to-days latency on non-McCarty-originated reports.
- **Not OSV/PURL-native.** Custom JSON + CPE-based STIX patterns mean normalization work; no native OSV-Scanner/Grype ingestion.
- **Key endpoints are paid, org-only, opaque-priced.** The free tier is real but limited (no bulk feed, no full IOCs, no STIX). Individuals cannot buy Pro.
- **Partial dependence on OSV/GHSA/Socket** undercuts its use as an independent corroborator for those exact records.
- **No first-party SDK** and no evidence of broad third-party tooling/community client libraries as of May 2026 — integration is DIY REST.
- **Terms of Use could not be machine-verified** (JS-rendered page); confirm exact acceptable/prohibited-use clauses in a browser before production use, especially regarding any data redistribution.
- **Key-person/early-stage risk.** OSM's differentiated value is tied substantially to McCarty and an informal researcher network; assess continuity/SLA before depending on it for blocking decisions.
- **Campaign figures vary by source.** OSM's own per-campaign counts (e.g., its Shai-Hulud tag, GlassWorm follow-up) differ from other vendors' (CISA, Datadog, Snyk, Socket, Aikido/Koi) because each tracks different slices and time windows — cross-reference before quoting any single number in incident reporting.
